    The commons registry

    Category: Reference. The authoritative list of shared packages, what each guarantees, and what may depend on what. For the why, see ARCHITECTURE.md.

    A "commons" is a versioned, independently-tested, independently-installable package that downstream code imports. Each is produced by exactly one delivery phase and must obey the dependency rule.

    Package Phase Language Stability Depends on Purpose
    schema 1 Python → TS experimental — Canonical domain model; the single source of truth
    engine 2 Python built schema Pure CPM + EVM computation, no I/O
    auth M1 Python built — Password hashing + JWT (self-hosted)
    persistence 3 Python built schema Storage-agnostic repositories
    services 3 Python built engine, persistence Orchestration / use-cases
    client 4 TypeScript in progress schema Generated types built (from schema, CI-checked) and now consumed by apps/web as a real workspace dependency; runtime API client wrapper still planned
    ui 5 TypeScript deferred (YAGNI) client, schema Design system built, but lives in apps/web/src/components — extracted here once a second consumer needs it; see packages/ui/README.md

    Rules that keep the registry honest

    • One producer per package. A package has a single owning phase and a single CODEOWNERS entry. Shared ownership means no ownership.
    • The public API is only what the README lists. Anything not documented in a package's ## Public API section is private and may change without a major version bump.
    • Dependencies only point inward. A package may import packages strictly below it in the table, never above, never an app. CI enforces this.
    • Shared logic moves down, never sideways. If two consumers need the same thing, it belongs in a commons below them — not duplicated, not imported across.

    See the how-to guide: docs/guides/adding-a-package.md. Adding a commons requires an ADR if it changes the dependency graph.
